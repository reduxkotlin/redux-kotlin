TaskFlow
Hi-Fi Design Spec

The visual system for TaskFlow, a Kanban sample app built on Compose Multiplatform with Material 3 Expressive. This document defines the design tokens, component library, and platform adaptation rules that the 7 screens are drawn from.

Direction A · Calm M3 Expressive Seed #4A3FB8 indigo Android · iOS · Web (wasmJs) · Desktop Compact · Medium · Expanded Light + Dark
Approval gate

This is Phase 1: foundations + component inventory + the Step 0 feasibility findings. Per the approved design, I'm holding here for your sign-off on tokens before drawing all 7 screens (compact + expanded, in device frames, fully red-lined) in Phase 2. Token values that need verification against the repo's resolved dependency graph are flagged inline.

What this system optimizes for

UX is P0 — the app has to feel smooth and alive. Material 3 Expressive is the vehicle: a higher-energy spring scheme (MotionScheme.expressive), shape-morph on interaction, emphasized type for hierarchy, and the new ButtonGroup / FloatingActionButtonMenu patterns. Every token below is chosen so the render-isolation story (a card move re-composes only two columns) is also visible — motion and color make the reactive updates legible.

Step 0

Feasibility spike

De-risking the UI before tokens are locked: can this repo's Compose 1.10.0 actually run experimental M3 Expressive, Coil 3 async images, and a markdown renderer on wasmJs? Below are recommended artifacts and fallbacks. Versions are drawn from current public releases and must be confirmed against the repo's resolved dependency graph — the JetBrains Material3 wrapper in particular is versioned independently of the core CMP plugin.

Verdict

All four capabilities are feasible today. Two carry conditions: (1) the exact Expressive-capable material3 artifact for CMP 1.10 must be pinned and gated with @OptIn(ExperimentalMaterial3ExpressiveApi::class); (2) Web requires a modern (Wasm-GC) browser, with a Kotlin/JS compatibility distribution as the floor. Neither blocks the design — Expressive composables degrade cleanly to their stable M3 counterparts where a build lacks them.

Foundations · 1

Color

A single seed (#4A3FB8) generates the full M3 tonal system. The role schemes below are authored to match Direction A — indigo primary, a rose tertiary for WIP/accents, warm-neutral surfaces. In code these come from expressiveLightColorScheme() / expressiveDarkColorScheme() seeded from the source color; the exact HCT-generated values should be reconciled with Material Theme Builder output.

Tonal palettes

Six tonal ramps (tones 0–100) underpin every role. Moving a card, dimming a filtered card, and the WIP badge all pull from these same ramps so state changes stay in-family.

Role scheme

Both modes ship. Surfaces use the M3 tonal container ladder (surfaceContainerLowest → …Highest) so cards, sheets and the board background separate by elevation tint rather than borders.

Semantic mapping

App concepts mapped onto roles so they re-theme automatically. Card labels are a fixed tonal set; WIP and sync states derive from container roles.

Foundations · 2

Typography

Type is Roboto Flex — the M3-native variable font, which gives the Expressive emphasized roles for free via the wght axis. The baseline scale is standard M3; the emphasized column raises weight (and tightens optical sizing) for screen titles, board names, selected nav, and primary actions. Specimens render live in Roboto Flex below.

Emphasis rule

Use exactly one emphasized element per visual group. Board name (Headline emphasized) + everything else baseline; never stack emphasized title and emphasized body in the same card.

Shape

The Expressive shape scale adds the increased and extra-extra-large steps that drive shape-morph: a card rounds up on press, the FAB morphs into the FAB-menu, a selected ButtonGroup segment expands. Corner values are the resting state; morph targets are noted per component in the inventory.

Foundations · 4

Spacing & layout

A 4 dp base unit. Touch targets never drop below 48 dp; the inter-column gutter and card padding carry most of the board's rhythm.

Window size classes

Adaptive layout keys off M3 window size classes. The board re-flows from a single paged column (compact) to all columns plus a persistent activity rail (expanded); the card detail moves from full-screen → bottom sheet → side sheet.

Platform adaptation — preview

iOS: respect safe-area insets; system back-swipe maps to dispatch(NavBack). Android: edge-to-edge, predictive-back progress drives the screen transition. Web: hover states + pointer affordances on cards/buttons, content capped at 1280 dp and centered. Full per-platform redlines land in Phase 2.

Foundations · 5

Elevation

M3 uses both a shadow and a tonal surface tint per level. Resting cards sit at Level 1; a dragged card lifts to Level 3 with a scale bump; the FAB-menu and dialogs own Level 3. In dark mode the tonal tint does the heavy lifting (shadows read weakly on dark surfaces).

Foundations · 6

Motion

Motion is governed by MotionScheme.expressive — six spring tokens split into spatial (anything that moves or resizes) and effects (color, opacity, elevation). Spatial springs carry a deliberate overshoot; effects springs never bounce. These are the token-level specs; the per-interaction choreography (card move, account switch, skeleton shimmer) is detailed in Phase 2.
