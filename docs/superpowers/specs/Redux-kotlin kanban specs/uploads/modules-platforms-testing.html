Design §3 — modules, platform entry points, testing

Last architecture section. Compose Multiplatform infra already exists in-repo (compose 1.10.0 + compose-compiler), so this is conventional wiring, not new ground.

Gradle module layout (under examples/, not published)
examples/taskflow/
├── composeApp/        ← KMP + Compose MP, ALL code + UI
│   plugins: convention.control,
│            kotlin("multiplatform"),
│            compose.multiplatform, compose.compiler
│   deps: redux-kotlin-bundle-compose
│   targets: androidTarget · jvm(desktop)
│            iosX64/Arm64/SimulatorArm64 (framework)
│            wasmJs { browser; executable }
├── androidApp/        ← com.android.application host
└── iosApp/            ← Xcode project embedding framework
                         (web = composeApp wasmJs + index.html;
                          desktop run for fast dev loop)

examples use convention.control → not published, no apiDump/apiCheck, and (no explicitApi()) no mandatory KDoc on every public. detekt formatting still applies tree-wide.

Platform entry points (thin — zero logic)
commonMain — App() composable + all Redux (stores, models, reducers, middleware, fake backend) + all screens.

Android — MainActivity { App() }
iOS — MainViewController() → SwiftUI ComposeView (mirrors counter/ios/Sources)
Web (wasmJs) — ComposeViewport(document.body){ App() } + index.html
Desktop (jvm) — application { Window { App() } } — dev-only, fastest iterate loop

All real code is shared; entry points only host App(). Same store wiring everywhere → proves true multiplatform.

Testing strategy (matches repo: commonTest default, jvmTest for JVM-only)

commonTest — the bulk (pure, fast, every target)

  • Routed reducers: each on<Action> transition
  • Undo/redo middleware: snapshot, undo, redo, cap, non-undoable ignored
  • Effects middleware: Request→Success/Failure with a fake backend + test dispatcher; optimistic revert on failure
  • Account isolation: actions on account A never touch account B's store; logout disposes only that store
  • Model inject/eject: enter/leave board adds/removes models, memory bounded

jvmTest — Compose UI (desktop, compose.uiTest)

  • Pattern already used by redux-kotlin-compose's jvmTest
  • A couple of high-value UI tests: the render-isolation proof (move a card → only 2 columns recompose), and account-switch restores each account's screen
  • Keep small — logic lives in commonTest

Native/iOS-sim tests trusted to CI per repo convention. Entry points have no logic → nothing to test there.
