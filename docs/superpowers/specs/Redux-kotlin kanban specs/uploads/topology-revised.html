Design §1+§2 (revised) — accounts make A and B compose

Multi-account is the legitimate registry use. So: B isolates accounts (one store per logged-in account), and A organizes each account's screens (single store + model inject/eject). Each API now earns its place honestly — no contrived cameo.

Two-layer topology
Root AppStore — one ConcurrentModelStore (app-global, always present)
AccountsModel · logged-in accounts + activeAccountId AppSettingsModel · theme, language, + fake-service knobs (latency, failure %, bot on/off) AuthFlowModel · login form + loading (transient)
▼ AccountRegistry = StoreRegistry<AccountId, ModelState> — one isolated store per account ▼
Account store: "ann" (active)
NavModel · this account's current screen (remembered!) SessionModel · profile BoardListModel · ann's boards injected on open: Board / Filter / Undo / Sync ActivityModel
Account store: "raj" fully isolated
own Nav (on Settings) · own boards · own undo switch accounts → each remembers its screen, like browser profiles logout → dispose this store from registry; ann untouched

Within an account: single store + dynamic model inject/eject bounds memory (A). Across accounts: registry gives true isolation + lifecycle (B). Switching = set activeAccountId; Compose binds registry.store(activeAccountId) via rememberStableStore.

Screen map (all Compose Multiplatform, nav = Redux state)
🔐 Login / Add account
root: AuthFlow 👥 Account switcher
root: Accounts 🗂 Board list
account: BoardList 📋 Board (kanban)
account: Board+Filter+Undo 🔎 Card detail
account: Nav.openCardId 👤 Profile / User
account: Session + logout ⚙️ Settings
root: AppSettings + fake-service knobs

Auth is fake: pick from a few seeded users (no real password); real/local auth can slot in behind the same actions later. Settings tunes the fake backend live (crank latency to 3s, set failure to 50%, pause the bot) — which doubles as a great way to see the loading/error/undo paths on demand.
