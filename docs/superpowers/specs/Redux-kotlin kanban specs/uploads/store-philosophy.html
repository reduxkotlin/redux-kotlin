Design §1 (revisited) — store philosophy

You're right that this is unsettled. I researched Redux core guidance, mobile (ReKotlin/ReSwift), and micro-frontend practice. Three coherent topologies below, with the evidence. Your "broadcast to all stores" idea is option C — analyzed honestly.

A — Single store + dynamic model injection ★ recommended default
One ConcurrentModelStore
always: Nav · Session · BoardList enter board → inject Board/Filter/Undo/Sync models (keyed by id) leave board → eject them → bounded memory

Mechanism: ModelState.with(...) + combineModelReducers + store.replaceReducer (verified to exist). This is redux-kotlin's version of Redux's injectReducer code-splitting pattern.

Pros

  • One action log, one subscribe — easiest to reason about
  • Directly fixes your memory con: eject offscreen state
  • Redux-canonical & mobile-canonical (ReKotlin = single store)
  • No "which store?" mental overhead

Cons

  • Under-uses StoreRegistry (a bundle API)
  • Inject/eject wiring is a bit advanced
  • No first-class model remove yet — rebuild via of(...)
B — Multi-store via registry, selectively coordinated
Root store + StoreRegistry<BoardId, ModelState>
per-board store created on navigate; disposed on leave cross-store = forward specific actions + read-only projection not broadcast — explicit, like Microsoft's redux-micro-frontend

Microsoft's production multi-store lib chose selective routing (register "global actions"), read-only cross-store views — explicitly NOT broadcast. That's the proven shape.

Pros

  • Uses the registry for its real purpose (isolation)
  • Per-board undo/filter naturally isolated
  • Realistic for independent "documents"

Cons

  • "Which store owns this action?" overhead — your stated con
  • Cross-store coordination is manual
  • No single DevTools/action log across stores
C — Multi-store + broadcast every action to all registered stores (your hybrid) — analyzed

Tempting: keep the "every reducer sees every action" simplicity of one store, but physically split. The problem: it inherits the cons of both.

Why it backfires

  • Kills the only reason to split. Isolation is the payoff of multi-store; broadcasting makes every store process irrelevant actions again — you pay the split cost for none of the benefit.
  • Reintroduces ordering coupling across stores (which store's reducer ran first?), the exact thing single-store avoids.
  • No major project does this. Microsoft's multi-store lib deliberately chose selective global-action routing over broadcast.
  • Harder to trace cause→effect than EITHER pure approach.

The legit version of the urge

  • If you want "every part sees every action" → that's option A (single store), and you bound memory with model ejection.
  • If you want isolation → that's option B with selective forwarding, not broadcast.
  • Broadcast is the unhappy middle; skip it.

My take: Make A the sample's primary architecture — it's the correct default, solves your memory concern with model inject/eject, and is the clearest teaching artifact. Then demonstrate B honestly in one contained place where isolation is genuinely wanted (e.g. an independent "scratch" board or a second workspace), with a short ADR explaining when to reach for the registry. That way the flagship sample teaches the decision, exercises the registry truthfully, and never ships the broadcast anti-pattern.
