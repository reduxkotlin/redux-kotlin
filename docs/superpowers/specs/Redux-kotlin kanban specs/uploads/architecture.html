Design §1 — State topology & store wiring

How state is split into stores and models. This is the spine of the showcase; it decides how honestly the registry + multimodel APIs get used. My recommendation is drawn solid; the alternative is noted below.

Recommended: root app store + StoreRegistry<BoardId, ModelState> (one store per board)
Root AppStore — createConcurrentModelStore { … }  multimodel + routing + concurrent
NavModel · route = BoardList | Board(id), open card id SessionModel · current user, online peers BoardListModel · board summaries (id, name, card count)
▼ navigating to a board lazily creates / looks up its store ▼
registry.getOrCreateConcurrentModelStore(boardId) { … }
Board store: "sprint-42"
BoardModel · columns + cards FilterModel · query, assignee UndoModel · history / future stacks SyncModel · loading / error / latency
Board store: "backlog" isolated
its own Board/Filter/Undo/Sync → each board has independent undo history & filter

Compose layer: rememberStableStore on root + active board store · columns use fieldStateOf(BoardModel::class){ it.columns[id] } · WIP/online badges use selectorState{…} · detail panel uses subscribeToModel/onModel

Why this (registry-per-board)

  • Uses StoreRegistry for its real purpose — keyed, lazily-created stores
  • Per-board isolation: undo history & filter don't leak across boards
  • Realistic: mirrors how multi-document apps actually shard state
  • Board store can be disposed when you leave (shows registry lifecycle)

Alternative: one global store

  • All boards as data inside one ModelState BoardsModel: Map<id, Board>
  • Simpler wiring, but registry becomes pointless → weaker showcase
  • Undo/filter must be keyed by board by hand inside one model
  • One giant reducer surface; less honest about real-world sharding

Cross-component-without-direct-access shows up here: a column never reads another column; the WIP badge never reads the detail panel; the activity feed never reads the board. They all subscribe to slices of the same store and re-render independently. That's the thing the demo is meant to prove.
